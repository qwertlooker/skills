# 华为胶片设计与制作

把“华为风”理解为一套沟通方法：先形成清晰结论，再用结构、证据与克制视觉降低理解成本。红白配色只是外观，不是核心。

## 先确定规范来源

按以下优先级执行：

1. 用户提供的正式PPTX模板或组织批准的母版；
2. 同一业务场景、同一业务单元的参考胶片；
3. 华为公开VI与联合品牌规范；
4. 华为公开业务胶片中可重复观察到的版式特征；
5. 本Skill的实践默认值。

有正式模板时，继承母版、Logo、字体、页脚、页码、密级、间距和版式，不要混入另一套模板。没有正式模板时，只能称为“华为风格”，不能称为“华为官方模板”。

## 使用配套资源

- HTML输出时复用 `resources/huawei-film-theme.css`。
- 需要网页胶片骨架时，从 `templates/slide-template.html` 开始修改。
- 需要理解输入到输出的转换时，阅读 `examples/example-input-output.md`。

## 工作流程

### 1. 定义汇报任务

先明确六项：

- **受众：** 谁参加，谁决策？
- **利益：** 对方更关注增长、效率、成本、风险、合规还是技术可行性？
- **认知：** 哪些背景可直接假设，哪些概念必须解释？
- **态度：** 支持、中立、质疑，还是倾向其他方案？
- **目标：** 汇报后需要对方理解、相信或决定什么？
- **约束：** 汇报时长、页数、输出格式、证据、模板和密级。

把目标写成可检验的一句话：

> 使[受众]在[会议/时间]内理解、认同或决定[具体结果]。

缺失信息只有在会显著改变故事线时才询问；其余采用低风险假设并明确标注。

### 2. 先搭故事线

先写整套材料的一句话结论，再选择一条主线：

- 问题 → 原因 → 方案 → 价值；
- 现状 → 差距 → 路径 → 结果；
- 趋势 → 挑战 → 应对 → 证明；
- 目标 → 架构 → 能力 → 路线图；
- 结论 → 证据 → 风险 → 决策。

不要机械添加公司介绍、行业趋势、架构、案例和路线图。只保留能推动受众决策的页面。

### 3. 形成页面观点

- 一页表达一个主要结论，通常用2–4组证据支撑。
- 标题写结论，不只写“行业趋势”“解决方案”“项目计划”等主题词。
- 页面内容遵循“观点—证据—业务含义”。
- 对决定性异议，提前安排证据；不要为了完整而罗列所有反方观点。
- 复杂架构旁增加一句“3秒摘要”，先说明它解决什么、关键机制是什么、首先看哪里。

### 4. 把技术指标翻译成业务语言

按三层表达，不虚构因果：

1. 保留有来源的技术或运营指标；
2. 解释它对实际流程产生什么影响；
3. 在证据支持时说明其决策含义。

示例：

- 弱表达：“推理吞吐提升2.3倍。”
- 有依据时的改写：“推理吞吐提升2.3倍，使现有集群能够覆盖预测峰值，暂不需要扩容。”

不能把基准测试直接换算成ROI、节省金额、质量提升或客户价值。估算、目标值和场景假设必须显式标注。

## 视觉规范

### 色彩

- 默认华为红使用公开VI中的 Pantone 185C：`RGB 199/0/11`、`#C7000B`。
- `RGB 153/0/0`、`#990000`仅作为旧培训模板兼容值；匹配继承模板时才使用。
- 日常会议优先白色或浅灰背景。深色版主要用于展厅、舞台、大屏或正式模板要求的场景。
- 红色只强调关键结论、核心数字、选中节点、重点线条或一条数据系列，避免大面积铺红。
- 单页通常不超过4种语义颜色；中性灰和必要的Logo原色不计入该限制。

### 字体与字号

- 默认中文使用 Microsoft YaHei（微软雅黑），默认英文和数字使用 Arial。
- 仅当正式模板明确要求且已安装授权字体时，才继承方正兰亭黑或 Huawei Sans；不得将其作为默认字体。
- 无模板时建议：封面38–44pt，页面标题30–32pt，正文18–22pt，图表标签14–18pt，来源9–11pt。
- 一页最多保留三级明显文字层级。正文不得为了塞内容缩小到16pt以下，应删减、重构或拆页。

### 版式

- 默认16:9，采用稳定页边距、统一对齐、直线条、浅分隔和扁平表达。
- 优先一个主体结构；只有真正并列的信息才使用2–4个模块。
- 避免大量圆角卡片、胶囊标签、强阴影、玻璃效果、3D图表和装饰性箭头。
- 封面和章节页可以使用一张强相关大图；常规内容页优先图表、架构和证据。
- 不把“每页≤35字”“留白必须≥30%”“Logo必须右上角”等经验值写成普遍硬规则。

### 图表与技术图

- 图表标题或页面标题必须说明分析结论。
- 红色突出一个关键系列或数据点，其余内容降为灰色。
- 标明单位、时间、口径、基准和来源；禁止3D、误导性截断和无必要双轴。
- 组网图和架构图使用一致的抽象符号、形状语义和阅读方向。
- 连线置于节点之后，减少交叉；信息过多时拆成“总览+细节”。
- 不要求所有流程图都出现反馈箭头，只有真实存在反馈机制时才绘制。

## 品牌与密级安全

- 只使用用户提供或从授权官方来源获得的Logo源文件。
- 不得用文字、CSS或自绘图形仿造“HUAWEI”Logo，也不得拉伸、改色、裁切、旋转或增加效果。
- 没有授权Logo时，省略Logo或保留明确的授权素材占位区域。
- 不得从参考胶片复制 `Huawei Confidential`、`Huawei Proprietary`、内部文档编号或限制分发声明。
- 密级、组织名和联合品牌关系必须由用户或正式模板明确提供。

## 输出路径

- **大纲/规划：** 先输出整套结论、故事线、逐页标题和每页1–2个关键证据。
- **PPTX：** 先检查当前环境是否提供原生演示文稿工具或Skill；可用时使用该能力并保持文字、表格、图表和简单形状可编辑，生成后逐页渲染检查。不可用时，不得声称已生成PPTX，应改为提供大纲、制作规范或HTML，并明确能力限制。
- **HTML：** 复用CSS主题和HTML模板，保留16:9、打印样式与响应式预览。
- **Marp/Slidev：** 仅在用户明确要求时输出，沿用相同内容和品牌规则。
- 不默认输出 `python-pptx` 代码；只有用户明确需要代码且运行环境允许时才采用。

## 场景适配

| 类型 | 推荐结构 | 重点 |
| --- | --- | --- |
| 领导汇报 | 结论 → 证据 → 风险 → 决策 | 选择、风险、责任与下一步 |
| 客户方案 | 场景 → 痛点 → 能力 → 证明 → 路径 | 客户结果优先于产品目录 |
| 技术交流 | 总览 → 机制 → 架构 → 细节 → 验证 | 先总后分、符号一致 |
| 进展汇报 | 结果 → 证据 → 问题 → 行动 → 请求 | 状态、责任、决策需求 |
| 演讲/展厅 | 一屏一意、少字、大字、强图 | 远距离可读性 |

## 质量检查

交付前执行三轮检查：

1. **逻辑：** 标题是否表达结论？页面内容是否证明标题？整套故事线是否完整、无重复？
2. **视觉：** 对齐、间距、字号、红色强调、图表语义和页面密度是否一致？
3. **完整性：** 数据、单位、日期、来源、Logo、密级、页脚和图表口径是否正确？是否存在虚构指标？

PPTX必须逐页渲染并检查文字折行、溢出、遮挡、低对比度、错误连线和不一致页脚。

## 资源校验

修改HTML或CSS后，在Skill目录执行：

```bash
node scripts/validate-template.mjs
```

校验失败时先修复路径、品牌安全、默认字体、页面数量或CSS结构，再交付。
